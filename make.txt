+, Cookie là gì ?
    -, Nơi lưu dữ liệu ở trình duyệt
    -, Tự động gửi kèm theo request
    -, Có httpOnly, secure.
    -, 👉 Cookie luôn được lưu trong trình duyệt (client).
    -, Lưu trong bộ nhớ / storage của browser
    -, Mỗi domain có cookie riêng
    -, Bạn có thể xem ở:
    -, DevTools → Application → Cookies
+, Session là gì ?
    -, Nơi lưu dữ liệu ở server
    -, Client chỉ dữ session_id
    -, Server map session_id -> server
+, JWT là gì ?
    -, Token stateless (Không có quốc tịch)
    -, Chứa thông tin user + chữ ký
    -, Server không cần lưu
+, flow (JWT + Cookie) : 
    Login
    → Server tạo JWT
    → Set Cookie HttpOnly
    → Request tự gửi token
    → Server verify JWT

3️⃣ Server Rendering (SSR / RSC) là gì?

Server tạo HTML trước rồi mới gửi về browser.

Ví dụ:

export default function Page() {
  return <h1>Hello Sơn</h1>
}

Luồng:

User request

Server chạy React

Server tạo HTML

Gửi HTML về

Browser nhận:

<h1>Hello Sơn</h1>

👉 Trang hiển thị ngay
👉 Không cần chờ JS
👉 SEO tốt

4️⃣ Client Rendering là gì?

Nếu bạn có:

'use client'

import { useState } from 'react'

Component này:

Không chạy trên server

Chỉ chạy trong browser

Có thể dùng useState, useEffect

5️⃣ Vậy “kết hợp” nghĩa là gì?

Ví dụ:

// Server Component
export default function Page() {
  return (
    <>
      <ServerData />
      <ClientButton />
    </>
  )
}

Trong đó:

'use client'
function ClientButton() {
  const [count, setCount] = useState(0)
  return <button onClick={() => setCount(count + 1)}>{count}</button>
}

Luồng xảy ra:

Bước 1: Server render trước

Server render:

ServerData → HTML thật

ClientButton → chỉ render HTML ban đầu

Server gửi HTML về.

Bước 2: Browser nhận HTML

UI hiển thị ngay.

Bước 3: JS load xong → Hydration

React gắn logic vào HTML đã có.

👉 Từ lúc này button mới click được.

6️⃣ Hydration là gì?

Hydration = React “gắn não” vào HTML đã render sẵn.

Nếu:

HTML server ≠ HTML client render lần đầu

→ Hydration mismatch

9️⃣ Tại sao auth hay bị chớp?

Vì bạn để phần quyết định auth ở client:

1. Static Rendering (Mặc định)
 => Toàn bộ nột dung trang web được tạo ra một lần duy nhất lúc bạn chạy lệnh NPM RUN BUILD
 
 +, Cách hoạt động: Nextjs tạo ra các file HTML tĩnh. Khi người dùng truy cập, server chỉ việc gửi file đó đi.
 +, Ưu điểm: Cực kỳ nhanh, tốt cho seo, chỉu tải tốt
 +, Nhược điểm: 
 +, Dùng khi nào: Trang Blog, Trang giới thiệu, trang tài liệu (Nhưng trang ít thay đổi nội dung)

2. Dynamic Rendering (Render theo yêu cầu)
=> Đây là kiểu "gọi món mới nấu". Trang web sẽ được tạo ra riêng biệt cho mỗi lần có người dùng truy cập (Request time).

  +, Cách hoạt động: Server sẽ chạy code, lấy dữ liệu mới nhất từ Database, sau đó mới trả về HTML cho trình duyệt. +, 
  +, Dấu hiệu nhận biết: Khi bạn dùng các hàm như cookies(), headers() hoặc fetch với tùy chọn { cache: 'no-store' }. +, 
  +, Ưu điểm: Dữ liệu luôn luôn là mới nhất, cá nhân hóa được nội dung (như giỏ hàng, thông tin user).  +, 
  +, Dùng khi nào: Trang quản trị (Dashboard), trang cá nhân, hoặc dữ liệu thay đổi liên tục theo giây.

3. Incremental Static Regeneration (ISR)
=> Đây là "phiên bản nâng cấp" của Static Rendering. Nó cho phép bạn cập nhật các trang tĩnh mà không cần build lại toàn bộ website.

  +, Cách hoạt động: Bạn thiết lập một khoảng thời gian (ví dụ 60 giây). Sau 60 giây, nếu có người truy cập, Next.js sẽ âm thầm tạo lại trang đó ở nền dưới với dữ liệu mới.
  +, Ưu điểm: Vừa nhanh như trang tĩnh, vừa có dữ liệu mới mà không tốn tài nguyên server như Dynamic.
  +, Dùng khi nào: Trang tin tức, danh sách sản phẩm, giá cả (thay đổi theo phút/giờ).
  +, Cú pháp: fetch('...', { next: { revalidate: 60 } })
